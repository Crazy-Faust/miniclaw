Abstract

miniclaw is a lightweight, local-first AI agent. It maps natural-language requests to a small set of safe tools, runs them on the user's machine, and writes every call to a SQLite audit log before the result is returned to the model.

The codebase is a pnpm workspace of 25 small packages. Every major subsystem — LLM provider, memory store, context strategy, skill, I/O, transport — lives behind an interface in @miniclaw/core, so each can be swapped or extended without touching the agent loop.

Three ways to run it: a single-process REPL, a long-lived daemon with chat attach over a Unix socket, or the same daemon answering Discord DMs.

Initial Situation & Problem

  • Cloud agents are opaque — no audit trail, data leaves the machine, vendor lock-in.
  • Most agent frameworks entangle the LLM, the tools, and the I/O — hard to extend safely.
  • Self-hosted alternatives skimp on the tool-call boundary: no sandbox, no allowlist, no confirmation tier.

Questions we set out to answer

  • Can a hobby-scale agent be provider-agnostic?
  • Can every tool call be auditable, by construction?
  • Can the tool boundary be safe by default (sandbox, allowlist, confirm)?
  • Can the same core drive a REPL, a daemon, and a Discord bot without rewrites?

Approach — architecture & key design decisions

miniclaw · Group #__

Key design decisions

1

Interface-first

Every subsystem behind a @miniclaw/core interface. Concrete impls are wired only in cli/ — anywhere else, importing a concrete class is a lint smell.

2

Audit before execute

Every tool call is written to audit_log before the result is handed back to the model. Even a misbehaving model leaves a trail.

3

Workspace sandbox

fs / shell / browser resolve every path against MINICLAW_WORKSPACE. Symlinks escaping the root are refused. Shell uses an allowlist with no shell interpolation and a 10s timeout.

4

Confirmation tier

Skills opt in with requiresConfirmation: true. Interactive mode prompts the user; one-shot mode fails closed. browser_click and browser_fill use this.

Results, Status & Reflection

Results / Demo

Current status

✓ Core interfaces · SQLite store · 3 LLM providers · REPL · one-shot
✓ Daemon + chat attach · cron persistence · Discord transport with pairing
▣ Vector memory · browser skills behind Playwright
☐ HTTP / SSE front-end polish · more transports (Slack, Matrix)

What worked

  • Small-package-per-interface kept blast radius tiny: adding Gemini was one new package, zero touches to the agent loop.
  • Audit-before-execute caught a real misbehaving prompt during dev — would have been invisible otherwise.
  • Pairing-code onboarding stopped strangers in field testing on a public Discord.

What didn't & what we'd change

  • Native build pain. better-sqlite3 on Windows is a porting tax — we tell users to use WSL.
  • Cron syntax is minimal (@every Ns|m|h|d). Full cron deferred — limits scheduling flexibility.
  • Vector memory not yet competitive with FTS5 at this scale. Would ship FTS-only next time and add embeddings later.
  • Order of work. Would ship the HTTP/SSE front-end before Discord, and pick one context strategy as canonical earlier.
