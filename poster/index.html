Abstract

miniclaw is a proof-of-concept desktop agent. A user writes a normal message, the agent chooses a skill, executes it inside a guarded workspace, and records the action before the model sees the result.

The prototype makes autonomy inspectable: every capability is typed, every action passes through the same loop, and long-running sessions can be traced through memory, audit logs, and transport channels.

Initial Situation / Problem

As LLM applications become more capable, they move from chat answers to delegated tasks. OpenClaw is one example: a personal agent that can act on a user's machine with increasing autonomy.

miniclaw builds a lightweight version of that idea. The focus is agent infrastructure: route language to tools, keep memory auditable, and execute actions safely.

Which tool should run? What should the agent remember? How does the agent reach out to the user?

Main Challenges

Intent-to-tool routing Plain language must become the right skill or chain of skills.
Memory management Retrieval should be fast, readable, and auditable.
Skill interfaces New capabilities need clear contracts, not core changes.
Security model Shell access, credentials, and untrusted inputs widen the attack surface.
Proactivity The agent needs rules for when to reach out and when to wait.

How Skills Work

01 AgentSkills format

Capabilities are described with SKILL.md files following the AgentSkills standard.

02 Catalog injection

The runtime discovers skills and injects a compact catalog of names and descriptions into the prompt.

03 Model chooses

The LLM selects a skill when the user's request matches a described capability.

04 Guarded execution

use_skill and run_skill_script execute through typed args, workspace limits, and audit logging.

Extensible tools New behavior is added by adding a skill package, not by changing the agent loop itself.

LLM Wiki Memory

Memory sources

write_memory stores new facts, notes, and observations as source material.

Small model manager

A smaller LLM organizes memories into wiki pages, links related pages, and keeps source rows traceable.

Prompt index

Only a compact index of relevant wiki pages is injected into the prompt, not the whole memory.

Explorable memory Based on the injected index, the model can decide when to call wiki_read or search_memory and inspect the memory in more detail.

Results / Code Grounding

What is actually implemented

miniclaw

Intent-to-tool routing

Routing is LLM-native tool calling, not local keyword rules. The agent sends the system prompt, recent history, and every registered tool spec each round.

  • WindowedContextManager tells the model when to use memory tools.
  • SkillRegistry.toolSpecs() converts Zod schemas to provider tool specs.
  • OpenAI, Anthropic, and Gemini adapters expose native tool/function APIs.
  • Multi-step chains run through the same agent loop with tool results fed back to the model.

AgentSkills

Skills follow the AgentSkills idea: a capability is documented, discoverable, and callable without changing the agent loop.

  • SKILL.md describes when and how the skill should be used.
  • The runtime discovers skills and injects a compact catalog into the prompt.
  • use_skill activates a skill; run_skill_script runs packaged handlers.
  • Built-ins cover filesystem, shell, web, database, memory, cron, sessions, todo, browser, and canvas.
  • Execution still goes through typed args, confirmations, workspace limits, and audit logging.

Security model

Sandboxing is application-level: commands and file operations are constrained to the configured workspace directory.

  • MINICLAW_WORKSPACE, default process.cwd()
  • Tool execution is sandboxed to the project directory instead of arbitrary machine access.
  • Filesystem rejects escapes, NUL bytes, and outside symlinks.
  • Writes and patches require confirmation and temp-file-plus-rename.
  • Shell uses argv, shell:false, allowlisted binaries, and guarded path args.
  • Optional high-security mode asks a small LLM to compare proposed tool calls with the initial prompt and flag malicious intent.
  • Rejected shell syntax includes backticks, $, $(, ||, and &&.
  • Web uses domain/private-host checks; SQL allows single SELECT/WITH.

Memory management

Long-term memory is managed as an LLM Wiki: raw writes become source material, then a small model organizes them into readable pages.

  • write_memory stores immutable raw memory rows in SQLite.
  • A maintenance worker uses a smaller LLM to upsert wiki pages, add links, and mark source rows.
  • Automatic retrieval injects only a compact memory index into the prompt.
  • The model can inspect details by calling wiki_read or search_memory.
  • Wiki pages are readable by the user through a local web interface.
  • Conversation history is still isolated by conv_id for daemon sessions and channels.

Proactivity

The model does not wake itself. Proactivity is scheduled prompts stored with channel, prompt, schedule, and next run time.

  • reminder_add and cron_add store jobs with the originating ctx.channel.
  • The daemon starts CronScheduler and handles background scheduled prompts.
  • Scheduler ticks immediately, then every 30s; @once removes before run, @every reschedules before run.
  • The gateway attaches the stored channel and sends the prompt through the normal agent loop.
  • Transport hooks can deliver trace.finalText back to the original channel.

Dreaming support

miniclaw also has a background reflection path: the dreaming runner can review recent activity and produce follow-up ideas.

  • @miniclaw/dreaming runs outside the foreground chat turn.
  • It uses the same agent and memory foundations as normal turns.
  • It supports reflective workflows such as reviewing recent chats and proposing next actions.
  • Dreaming is separate from cron: cron schedules prompts; dreaming summarizes and reflects.
