Abstract

miniclaw is a proof-of-concept desktop agent. A user writes a normal message, the agent chooses a skill, executes it inside a guarded workspace, and records the action before the model sees the result.

The prototype makes autonomy inspectable: every capability is typed, every action passes through the same loop, and long-running sessions can be traced through memory, audit logs, and transport channels.

Initial Situation / Problem

As LLM applications become more capable, they move from chat answers to delegated tasks. OpenClaw is one example: a personal agent that can act on a user's machine with increasing autonomy.

miniclaw builds a lightweight version of that idea. The focus is agent infrastructure: route language to tools, keep memory auditable, and execute actions safely.

Which tool should run? What should the agent remember? How does the agent reach out to the user?

Main Challenges

Intent-to-tool routing Plain language must become the right skill or chain of skills.
Memory management Retrieval should be fast, readable, and auditable.
Skill interfaces New capabilities need clear contracts, not core changes.
Security model Shell access, credentials, and untrusted inputs widen the attack surface.
Proactivity The agent needs rules for when to reach out and when to wait.

Key Design Decisions

1

Model routes tools

No custom classifier: tool specs, schemas, system prompt, and history are sent to the provider each round.

2

Memory is layered

Long-term memories are separate from per-session turn history, then merged by the context manager.

3

Skills are contracts

A skill is name, description, Zod parameters, optional confirmation, and execute(args, ctx).

4

Sandbox at boundaries

Security is per skill/tool: filesystem, shell, SQL, web, and confirmations each enforce their own guard.

Results / Code Grounding

What is actually implemented

miniclaw

Intent-to-tool routing

Routing is LLM-native tool calling, not local keyword rules. The agent sends the system prompt, recent history, and every registered tool spec each round.

  • WindowedContextManager tells the model when to use memory tools.
  • SkillRegistry.toolSpecs() converts Zod schemas to provider tool specs.
  • OpenAI, Anthropic, and Gemini adapters expose native tool/function APIs.
  • Multi-step chains run through the same loop, capped by MAX_ROUNDS = 6.

Skill interface

Adding a capability means exporting one object, then registering it once. The context carries memory, audit, DB path, channel, workspace root, and streaming output.

  • name, description
  • parameters: Zod schema
  • requiresConfirmation?
  • execute(args, ctx)
  • Dispatch validates args, confirms risky skills, executes, audits, returns result.

Security model

Sandboxing is application-level and workspace-root based.

  • MINICLAW_WORKSPACE, default process.cwd()
  • Filesystem rejects escapes, NUL bytes, and outside symlinks.
  • Writes and patches require confirmation and temp-file-plus-rename.
  • Shell uses argv, shell:false, allowlisted binaries, and guarded path args.
  • Rejected shell syntax includes backticks, $, $(, ||, and &&.
  • Web uses domain/private-host checks; SQL allows single SELECT/WITH.

Memory management

Two layers: long-term facts/preferences/tasks, plus conversation history per session.

  • Memory contract: add, search, listRecent.
  • SQLite stores memories with FTS5 search by rank.
  • Each turn injects up to 5 memory index entries and last 12 messages.
  • Messages are stored by conv_id; daemon channels get isolated sessions.
  • In-memory and vector stores exist; vector search is still experimental.
  • Compaction exists, but default behavior is windowing.

Proactivity

The model does not wake itself. Proactivity is scheduled prompts stored with channel, prompt, schedule, and next run time.

  • reminder_add and cron_add store jobs with the originating ctx.channel.
  • The daemon starts CronScheduler and handles background scheduled prompts.
  • Scheduler ticks immediately, then every 30s; @once removes before run, @every reschedules before run.
  • The gateway attaches the stored channel and sends the prompt through the normal agent loop.
  • Transport hooks can deliver trace.finalText back to the original channel.
Opt-in / partial
  • Playwright browser skills require optional dependency
  • Discord reminders work through daemon transport
  • compaction and vector semantic memory are not the default path
Possible improvements
  • move beyond app-level sandboxing for high-risk actions
  • stronger prompt-injection and credential scopes
  • Use a small LLM to cluster, rewrite, and retire SQLite memories.
  • Add markdown-skills / agentskills support.
  • Run nightly dreaming: review recent chats and propose next actions.
  • Define external tool adapters before expanding the skill set.
